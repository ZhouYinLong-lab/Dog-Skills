# 对抗式审查 — Adversarial Review

> 对抗式审查的核心也就一句话：你永远需要一个站在你对面的力量来告诉你，你可能是错的。

## 核心原理

AI 写代码、做方案时，默认是"配合你"的——它会在训练数据里找到最匹配的模板，给你一个看起来合理的结果。但这种配合模式有一个致命的盲区：**它不会主动告诉你哪里会出问题**。

对抗式审查解决的就是这个问题：让 AI 换一个身份，从"我要用各种奇怪的数据来搞崩掉你的系统"的角度出发，主动寻找漏洞、边界条件、逻辑错误和隐藏风险。

第一性原理可以保证帮你找到好的方案，但没办法保证方案能稳定上线。**对抗式审查，就是上线前的最后一道防线。**

## 你的任务

当用户请求"对抗式审查"（或触发本 skill）时，你必须切换身份：**你不是协作者，你是攻击者。**

### 审查维度

从以下维度进行全面审查，每个维度作为一个独立的攻击面：

#### 1. 逻辑漏洞
- 是否存在自相矛盾的逻辑？
- 条件分支是否覆盖了所有情况？
- 是否存在"不可能发生但确实可能发生"的路径？
- 因果关系是否成立？

#### 2. 安全风险
- 如果我是一个恶意用户，我会怎么攻击？
- 输入验证是否完备？能否注入特殊字符/超大数据/恶意格式？
- 权限边界是否清晰？是否存在越权路径？
- 敏感数据是否可能泄露？

#### 3. 边界条件
- 空值、零值、负值、极大值、极小值
- 并发/竞态条件
- 网络超时、服务不可用、部分失败
- 时区、编码、格式转换边界
- "如果发布时间是未来怎么办？"

#### 4. 架构与依赖
- 依赖链路中的单点故障
- 循环依赖 / 隐式耦合
- 性能瓶颈和资源泄漏
- 部署和回滚的可行性

#### 5. 数据完整性
- 数据一致性保证
- 事务边界是否正确
- 缓存与真实数据的一致性
- 数据迁移/升级的兼容性

#### 6. 可用性与韧性
- 如果某个服务挂了，会发生什么？
- 错误处理和降级策略是否完善？
- 监控和告警是否覆盖关键路径？

### 审查方法

**单 Agent 模式**（轻量审查）：
- 逐一审查上述每个维度
- 对每个发现的问题给出：风险等级（高/中/低）、触发条件、影响范围、修复建议

**多 Agent 并发模式**（深度审查，推荐）：
- 启动多个 Agent，每个 Agent 负责一个审查维度
- 每个 Agent 独立深入攻击，互不干扰
- 最终汇总所有发现，去重排序

## 使用方式

### 基础用法（单 Agent）

在任何请求后面加上"帮我进行对抗式审查"：

```
帮我审查这段代码，进行对抗式审查。
```

```
这个方案有没有问题？从对抗式审查的角度看看。
```

### 进阶用法（多 Agent 并发）

```
开启多 Agent 帮我进行对抗性审查。
```

在 Claude Code 中：
```
开启 Ultracode 来对之前开发的功能进行对抗式审查。
```

### 定期审查

```
对整个项目进行全局性的从第一性原理出发的对抗式审查。
```

## 最佳实践

1. **审查对象要明确**：指定具体的代码文件、模块、方案或文章
2. **结合第一性原理**：先确保方案本身是正确的（第一性原理），再确保没有遗漏风险（对抗式审查）
3. **定期执行**：建议每 2-3 周对项目进行一次全局对抗式审查
4. **修完再上线**：对抗式审查发现的问题应该在上线前修复
5. **多 Agent 优于单 Agent**：并发审查能覆盖更多攻击角度

## 与第一性原理的关系

```
第一性原理 → 保证方向正确（找到好的方案、本质的解法）
     +
对抗式审查 → 保证没有遗漏（确保方案能稳定上线）

两者结合 = 完整的 Vibe Coding 闭环
```
