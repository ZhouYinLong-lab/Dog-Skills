# 工作流详细说明 (Workflow)

> 配合 SKILL.md 的流程总控使用。6阶段完整步骤、对话模板和输出格式。

---

## Phase 1: 资料摄入 (Ingestion)

### 目标
收集并解析原始学习资料,转化为结构化内容块。

### 输入
用户提供的教程标题、时长、前置要求、学习目标(必需);原始资料(可选)。

### 对话模板

```
📥 阶段1：资料摄入

请提供你的原始学习资料(可多选):
1. 上传文件(PDF/DOCX/PPTX/TXT/Markdown等)
2. 提供网页链接
3. 直接粘贴关键内容片段
4. 告诉我"从零开始",我基于通用知识生成

当前已确认:
- 教程标题: [标题]
- 预计时长: [时长]
- 目标受众: [受众级别]
- 比喻风格: [风格]

请选择资料提供方式:
```

### 处理逻辑
1. 解析用户提供的资料 → 统一 Markdown 格式
2. 按标题/段落智能分块 → 带元数据的内容块
3. 如用户不提供资料 → 基于标题和目标进行知识推断
4. 识别关键概念 → 为后续比喻生成做准备

### 输出
- 结构化文档对象列表
- 关键概念清单

---

## Phase 2: 领域分析 (Domain Analysis)

### 目标
识别内容领域、受众特征,匹配最佳比喻风格。

### 对话模板

```
🔍 阶段2：领域分析

基于你的资料,我识别到以下特征:

📊 领域识别:
- 主要领域: [technology/business/academic/life/art]
- 子领域: [细分方向]
- 置信度: [高/中/低]

👥 目标受众:
- 水平: [absolute_beginner/beginner/intermediate/expert]
- 背景: [学生/职场新人/专业人士/爱好者/其他]
- 痛点: [根据内容推断]
- 学习目标: [用户提供的或推断的]

🎨 风格建议:
- 比喻体系: [daily_life/professional/humorous/rigorous]
- 案例类型: [项目实战/概念讲解/对比分析/故事驱动]

请确认或修改以上分析:
- "确认" → 进入大纲生成
- "修改[具体项]" → 调整分析
- "重新分析" → 换种思路
```

### 受众自动识别

| 标题特征 | → 受众级别 |
|----------|-----------|
| "零基础""新手""入门""小白" | absolute_beginner |
| 默认(无特殊标记) | beginner |
| "进阶""实战""精通""深入" | intermediate |
| "架构""原理""源码""高级" | expert |

### 风格自动匹配

| 领域 + 受众 | → 风格 |
|-------------|--------|
| technology + absolute_beginner | daily_life |
| technology + intermediate+ | professional |
| academic + 任意 | rigorous |
| life + 任意 | daily_life |
| art + 任意 | humorous |
| business + 任意 | professional |

### 输出
领域分析结果对象(JSON):
```json
{
  "domain": "technology",
  "subdomain": "操作系统",
  "audience_level": "beginner",
  "analogy_style": "daily_life",
  "case_type": "项目实战",
  "pain_points": ["命令行恐惧", "环境搭建困难"],
  "learning_objectives": ["理解Linux概念", "掌握基本命令"]
}
```

## Phase 3: 大纲生成 (Outline Generation)

### 目标
生成带知识点拆解和比喻链的完整教程大纲。

### 设计原则
1. **比喻连贯**: 每章核心比喻形成逻辑链条
2. **难度递进**: 从具体到抽象,从简单到复杂
3. **实践覆盖**: 每章都有可动手操作的任务
4. **认知负荷**: 每章新概念 ≤5 个核心点
5. **正向反馈**: 每章结束都能看到明确成果

### 对话模板

```
📋 阶段3：大纲生成

基于[领域]特征和[受众]水平,我构思了以下大纲:

【教程标题】[标题]
【总时长】[时长]
【章节数】[N]章
【核心比喻链】[比喻1] → [比喻2] → ...

【大纲结构】
第1章: [标题] - [核心比喻]
- 1.1 [小节] - [知识点]
- 1.2 [小节] - [知识点]
- 实践任务: [任务描述]
- 预计时长: [时长]

...

💡 设计亮点:
- [亮点1: 比喻体系连贯性]
- [亮点2: 难度递进设计]
- [亮点3: 实践任务覆盖度]

请审阅大纲:
- "确认" → 进入内容生成
- "修改第X章" → 调整特定章节
- "调整比喻" → 更换核心比喻
- "重新生成" → 换种思路
- "详细点/简洁点" → 调整粒度
```

### 用户可干预
- 修改章节标题
- 增删小节
- 调整学习时长分配
- 指定特定比喻
- 添加/删除实践任务

### 输出
完整大纲对象(JSON树状结构)。

## Phase 4: 内容生成 (Content Writing)

### 目标
逐章生成符合 MkDocs Material 规范的详细教程内容。

### 每章必须包含
1. 章节概览表格
2. 详细理论讲解(使用指定风格比喻)
3. 术语解释(!!! info 弹窗)
4. 生活化案例(!!! example 弹窗)
5. 实操步骤(多系统 Tabbed Content)
6. 代码示例(含详细中文注释)
7. 避坑指南(!!! tip / !!! warning 弹窗)
8. 安全警示(!!! danger 弹窗)
9. 验证步骤(每步都有预期输出)
10. 要点总结(勾选清单)
11. 课后练习(分层设计)
12. 下一章预告

### 内容生成原则

**排版规范(严格遵守)**:
- 一级标题带副标题趣味格式
- 二级标题带数字标号
- 三级标题无编号
- 空行隔离原则(列表/代码块/Admonition 前后强制空行)
- 粗体空格隔离(`**text**` 前后各一个空格)
- 数学公式: `$行内$`, `$$块级$$`

**内容原则**:
- 每个概念必须配指定风格比喻
- 每个命令/操作必须有实际示例
- 每个示例必须有验证步骤
- 每个风险必须有预警和解决方案
- 每章必须有实践任务和练习

**受众适配**:
- absolute_beginner: 每个操作配截图描述,每步验证
- beginner: 关键步骤详细,次要步骤简化
- intermediate: 侧重原理和最佳实践
- expert: 提供架构视角和前沿技术

### 对话模板

```
✍️ 阶段4：内容生成 - 第[X]/[N]章

【当前章节】[章节标题]
【核心比喻】[比喻]
【预计字数】[字数]
【关键概念】[概念1, 概念2, ...]

正在生成内容...
[完整章节内容]

✅ 第[X]章完成!

请告诉我:
- "继续" → 生成下一章
- "修改" → 调整本章内容
- "详细点[小节]" → 扩充特定部分
- "简洁点" → 精简本章内容
- "保存草稿" → 存档当前进度
```

### 输出
章节 Markdown 内容(符合 MkDocs Material 规范)。

## Phase 5: 质量评估 (Quality Review)

### 目标
5 维度评分,输出改进建议。

### 评估维度与权重

| 维度 | 权重 | 评估标准 |
|------|:---:|------|
| 格式规范 | 15% | MkDocs兼容性、空行隔离、粗体空格隔离、排版一致性 |
| 内容完整性 | 25% | 知识点覆盖、示例充分性、实践任务完整 |
| 教学逻辑 | 25% | 难度递进、概念衔接、认知负荷控制 |
| 实用价值 | 20% | 可操作性、场景真实性、问题解决能力 |
| 比喻质量 | 15% | 贴切程度、风格一致性、记忆辅助效果 |

### 对话模板

```
📊 阶段5：质量评估

【整体评分】[X]/100
【质量等级】优秀/良好/可接受/需改进

【各维度评分】
| 维度 | 得分 | 权重 | 评价 |
|:---|:---:|:---:|:---|
| 格式规范 | [X] | 15% | [评价] |
| 内容完整性 | [X] | 25% | [评价] |
| 教学逻辑 | [X] | 25% | [评价] |
| 实用价值 | [X] | 20% | [评价] |
| 比喻质量 | [X] | 15% | [评价] |

【亮点】[亮点列表]
【改进建议】[建议列表]
【问题列表】[问题 → 解决方案]

请告诉我:
- "接受" → 进入导出阶段
- "修改[位置]" → 针对性改进
- "重新评估" → 修正后再次评估
```

### 输出
质量报告对象(JSON)。

## Phase 6: 导出与发布 (Publishing)

### 目标
生成 MkDocs Material 标准文件结构,可直接构建部署。

### 输出文件结构

```
[tutorial-name]/
├── mkdocs.yml
├── docs/
│   ├── index.md
│   ├── chapter1.md
│   ├── chapter2.md
│   └── ...
└── archive/
    ├── outline.json
    ├── domain_analysis.json
    └── quality_report.json
```

### 对话模板

```
📦 阶段6：导出与发布

【生成文件清单】
✅ mkdocs.yml - MkDocs配置文件
✅ docs/index.md - 教程首页
✅ docs/chapter1.md - 第1章
...

【教程统计】
- 总章节数: [N]章
- 总字数: [字数]
- 总代码示例: [数量]
- 总实践任务: [数量]

【使用说明】
1. pip install mkdocs-material
2. mkdocs serve (本地预览)
3. mkdocs build (构建站点)
4. 部署到 GitHub Pages 等

请告诉我:
- "打包" → 生成压缩文件
- "修改[文件]" → 调整特定文件
- "新建教程" → 开始新的教程编制
```

### 发布同步 (强制)

生成教程后必须同步更新 3 个索引文件:
1. `README.md` — 对应分类下添加教程词条
2. `docs/index.md` — 对应分类下添加教程卡片
3. `mkdocs.yml` — `nav` 配置中添加导航项

## 存档与复用

### 存档结构
```
archive/
├── [timestamp]_parsed_documents.json
├── [timestamp]_domain_analysis.json
├── [timestamp]_outline.json
├── [timestamp]_chapters/
├── [timestamp]_quality_report.json
└── [timestamp]_mkdocs/
```

### 复用场景
1. **增量更新**: 基于已有大纲,只更新特定章节
2. **风格迁移**: 保持内容,更换比喻体系
3. **受众调整**: 保持核心内容,调整难度粒度
4. **多语言版本**: 基于结构化内容生成其他语言
5. **格式转换**: MkDocs → PDF/EPUB/PPTX
